## AGENT 1 — DataSentinel  (the inspector)

**Instruction.** Monitor incoming sensor windows for each asset. Detect and *type* abnormal readings. Report them; do not fix them.

**Rule (hard).**
- Flags only — **never modifies, drops, or imputes data.**
- Non-autonomous. No downstream action on its own.

**Engine.**
- Per-signal **Z-score** for univariate spikes.
- **Mahalanobis distance** across signals for multivariate anomalies (a combination that's individually-normal but jointly-impossible).
- Severity = function of how far beyond threshold.
- LLM (optional, Llama 3.2 3B) only to name the anomaly type in plain words.

**Output.** `DataQualityReport(asset_id, signal_id, anomaly_type, severity, ts)`

## AGENT 2 — AssetHealthOracle  (the doctor)

**Instruction.** Continuously score each asset's health (0–100) and estimate Remaining Useful Life (RUL) in days, with a confidence interval and a plain-English physics explanation.

**Rule (hard).**
- Scores and reports only. **Non-autonomous.**
- Every number carries a confidence interval. Never report RUL without `ci_95`.

**Engine.**
- **Weibull degradation model** (analytical, always works):
  `H(t) = 100 · exp(−λ·t^β)` × temperature (Arrhenius) × load corrections.
- λ, β pre-calibrated from CMAPSS/PRONOSTIA per asset class.
- RUL from inverting the Weibull curve to the failure threshold; CI via bootstrap (n=1000).
- **Optional PINN correction** (`δ_LSTM`) if trained — otherwise skipped with no visible difference.

**Output.** `AssetHealthReport(asset_id, health_score, rul_days, ci_95, physics_text)`

## AGENT 3 — GötzeEngine  ⭐ (the coach) — the differentiator

**Instruction.** Given a stressed asset, evaluate *every* candidate human intervention, score each by impact, and surface the **single best action** with a short, defensible reason. Present runner-up + the gap so the human sees it wasn't arbitrary.

**Rule (hard).**
- **REQUIRES human approval** before any action is recorded as taken. Absolutely non-autonomous.
- Must always show: the chosen action, its score, the runner-up, the gap, and a reason.
- Must never recommend an action whose `SafetyRiskDelta` exceeds the safety ceiling (hard veto).

**Engine — the Intervention Impact Score (IIS):**
```
IIS(i) = 0.35·ΔP_failure(i)      # how much it cuts failure probability
       + 0.25·ΔDowntimeCost(i)   # how much downtime cost it avoids
       + 0.20·Feasibility(i)     # crew + parts available now?
       + 0.15·HistoricalSuccess(i)
       − 0.05·SafetyRiskDelta(i) # penalize added risk
```
- All terms normalized to [0,1] before weighting → IIS is comparable across actions.
- **Demo: fixed weights.** Production: weights self-calibrate from `FeedbackLoop` outcomes (narrative only).
- Reason narrative generated by Groq Llama 3.3 70B (fallback: templated string per action).

**Output.** `GötzeDecision(top_intervention, iis_score, runner_up, iis_gap, narrative, confidence, requires_human_approval=True)`

## AGENT 4 — RootCauseAnalyst  (the detective)

**Instruction.** Explain *why* the anomaly happened. Search maintenance manuals, fault logs, and SOPs; build a causal chain; cite every claim; surface similar past faults.

**Rule (hard).**
- Every causal step must carry a citation or be marked low-confidence.
- Never fabricate a manual reference. No source → say "uncertain."

**Engine.**
- **RAG**: embed query with `all-MiniLM-L6-v2`, retrieve top-k from ChromaDB.
- LLM (DeepSeek R1 or Groq Llama 3.3 70B) assembles the causal chain from retrieved chunks only.
- Similarity search over past fault records for "we've seen this before."

**Output.** `CausalChain(anomaly_id, steps, citations, confidence, similar_past_faults)`
